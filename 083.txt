**مرحله ۱ - درک مسئله و تعیین محدوده طراحی**  
محدودسازی نرخ (Rate Limiting) را می‌توان با الگوریتم‌های مختلفی پیاده‌سازی کرد که هرکدام مزایا و معایب خود را دارند. گفت‌وگوی بین مصاحبه‌کننده و کاندیدا به شفاف‌سازی نوع محدودکننده‌ای که می‌خواهیم بسازیم کمک می‌کند.  

*کاندیدا:*  
چه نوع محدودکننده‌ای قرار است طراحی کنیم؟ آیا یک محدودکننده سمت کلاینت است یا محدودکننده API سمت سرور؟  

*مصاحبه‌کننده:*  
سوال خوبی است. تمرکز ما روی محدودکننده API سمت سرور است.  

*کاندیدا:*  
آیا محدودکننده درخواست‌های API را بر اساس IP، شناسه کاربر (User ID) یا سایر ویژگی‌ها تنظیم می‌کند؟  

*مصاحبه‌کننده:*  
محدودکننده باید به اندازه‌ای انعطاف‌پذیر باشد که از مجموعه‌های مختلفی از قوانین تنظیم نرخ پشتیبانی کند.  

*کاندیدا:*  
مقیاس سیستم چقدر است؟ آیا برای یک استارتاپ طراحی می‌شود یا یک شرکت بزرگ با پایگاه کاربری گسترده؟  

*مصاحبه‌کننده:*  
سیستم باید بتواند حجم بالایی از درخواست‌ها را مدیریت کند.  

*کاندیدا:*  
آیا سیستم در یک محیط توزیع‌شده کار می‌کند؟  

*مصاحبه‌کننده:*  
بله.  

*کاندیدا:*  
آیا محدودکننده یک سرویس جداگانه است یا باید در کد برنامه پیاده‌سازی شود؟  

*مصاحبه‌کننده:*  
این یک تصمیم طراحی است که به عهده شماست.  

*کاندیدا:*  
آیا باید به کاربرانی که درخواست‌هایشان محدود شده است اطلاع دهیم؟  

*مصاحبه‌کننده:*  
بله.  

---  
**نیازمندی‌ها**  
در اینجا خلاصه‌ای از نیازمندی‌های سیستم آورده شده است:  

• محدودسازی دقیق درخواست‌های بیش از حد.  
• تأخیر کم: محدودکننده نباید زمان پاسخ HTTP را کند کند.  
• استفاده از حداقل میزان حافظه ممکن.  
• محدودسازی نرخ توزیع‌شده: محدودکننده باید بین چندین سرور یا فرآیند قابل اشتراک باشد.  
• مدیریت استثناها: نمایش خطاهای واضح به کاربران زمانی که درخواست‌هایشان محدود می‌شود.  
• تحمل خطای بالا: اگر مشکلی در محدودکننده پیش آید (مثلاً سرور کش آفلاین شود)، کل سیستم تحت تأثیر قرار نگیرد.
