**مرحله ۲ - ارائه طرح کلی و جلب موافقت**  
در این بخش، به بررسی نقاط پایانی API، تغییر مسیر URL و فرآیندهای کوتاه‌سازی URL می‌پردازیم.  

---  

**نقاط پایانی API**  
نقاط پایانی API ارتباط بین کلاینت‌ها و سرورها را تسهیل می‌کنند. ما این APIها را به سبک REST طراحی خواهیم کرد. اگر با APIهای RESTful آشنا نیستید، می‌توانید از منابع خارجی مانند منبع ذکرشده در [1] استفاده کنید.  

یک سرویس کوتاه‌کننده URL عمدتاً به دو نقطه پایانی API نیاز دارد:  

1. **کوتاه‌سازی URL**  
   برای ایجاد یک URL کوتاه جدید، کلاینت یک درخواست POST ارسال می‌کند که شامل یک پارامتر است: URL اصلی بلند. این API به شکل زیر است:  

   *POST api/v1/data/shorten*  
   • پارامتر درخواست: `{longUrl: longURLString}`  
   • خروجی: `shortURL`  

2. **تغییر مسیر URL**  
   برای هدایت یک URL کوتاه به URL بلند متناظر، کلاینت یک درخواست GET ارسال می‌کند. این API به شکل زیر است:  

   *GET api/v1/shortUrl*  
   • خروجی: `longURL` برای تغییر مسیر HTTP  

---  

**تغییر مسیر URL**  
شکل ۸-۱ نشان می‌دهد که وقتی یک tinyurl را در مرورگر وارد می‌کنید چه اتفاقی می‌افتد. هنگامی که سرور درخواست tinyurl را دریافت می‌کند، URL کوتاه را با تغییر مسیر ۳۰۱ به URL بلند تبدیل می‌کند. ارتباط دقیق بین کلاینت‌ها و سرورها در شکل ۸-۲ نمایش داده شده است.
